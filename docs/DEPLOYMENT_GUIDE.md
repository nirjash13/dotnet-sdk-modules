# SaaS-Builder SDK Deployment Guide

## 1. Deployment Topology Overview

### Single-Region, Single-Stamp (Default — Phase 1)

```
┌─────────────────────────────────────────┐
│  Load Balancer / Reverse Proxy (Nginx)  │
└────────────────┬────────────────────────┘
                 │
    ┌────────────┴────────────┐
    │                         │
┌───▼──────┐         ┌───────▼──┐
│  App     │         │  App     │
│  Pod 1   │         │  Pod 2   │
└───┬──────┘         └────┬─────┘
    │                     │
    └──────────┬──────────┘
               │
    ┌──────────▼──────────┐
    │   Postgres 16+      │
    │  (single pool)      │
    │  with RLS enabled   │
    └─────────────────────┘
               │
    ┌──────────▼──────────┐
    │    RabbitMQ         │
    │  (optional, Bus)    │
    └─────────────────────┘
```

All tenants share a single Postgres database. Data isolation via RLS + EF Core filters.

### Multi-Region, Multi-Stamp (Deferred — Phase 3)

**Status: Deferred** — Phase 3 will introduce `SiloedStamp` mode and `IStampRouter` for region-per-tenant routing.

### Pool vs. Silo Isolation Modes (Phase 3)

| Mode | Database | RLS | Scaling | Cost | Data Residency |
|------|----------|-----|---------|------|---|
| `PoolShared` | Shared | No | High (noisy neighbor risk) | $$$ | Not applicable |
| `PoolWithRls` (default) | Shared | Yes | High | $$$ | Shared region |
| `SiloedSchema` | Shared | Per-schema | Medium | $$$ | Shared region |
| `SiloedDatabase` | Per-tenant | Yes | Low | $$$$ | Shared region per tenant |
| `SiloedStamp` | Per-tenant per-stamp | Yes | Low | $$$$$ | Region per tenant |

**Phase 1 supports:** `PoolWithRls` only.

## 2. Prerequisites

### Required

- **Postgres 16+** (tested; 14+ may work)
  - RLS must be supported (built-in for PG 9.5+)
  - `uuid-ossp` extension (generated by migrations)
- **.NET 10** runtime on deployment host
- **Docker** (if using Docker Compose / Kubernetes)

### Optional

- **RabbitMQ 3.13+** (only if using `SaasTransport.Bus` for out-of-process messaging)
- **Redis 7+** (only if using SignalR backplane for realtime, Phase 5)
- **OpenTelemetry Collector** (if sending traces/metrics to central backend)
- **Cloud secrets manager** (Azure Key Vault, AWS Secrets Manager, GCP Secret Manager)

## 3. Environment Configuration

### Required Environment Variables

| Variable | Purpose | Example |
|----------|---------|---------|
| `ConnectionStrings__SaasBuilder` | Postgres connection | `Server=db.example.com;Database=SaasBuilder;Username=saas;Password=...` |
| `Identity__Authority` | OIDC provider (your app or external IdP) | `https://auth.yourapp.com` |
| `Identity__Audience` | JWT audience claim | `api.yourapp.com` |

### Optional Environment Variables (Per Module)

| Variable | Purpose | Used By | Example |
|----------|---------|---------|---------|
| `Dispatch__Transport` | Mediator (inproc) or Bus (rabbitmq) | Host | `inproc` or `bus` |
| `RabbitMq__Host` | RabbitMQ AMQP endpoint | Bus transport | `rabbitmq.example.com` |
| `RabbitMq__Port` | RabbitMQ port | Bus transport | `5672` |
| `RabbitMq__Username` | RabbitMq user | Bus transport | `guest` |
| `RabbitMq__Password` | RabbitMq password | Bus transport | (secret) |
| `Otel__Endpoint` | OpenTelemetry Collector OTLP endpoint | Observability | `http://otel-collector.example.com:4317` |
| `Otel__Enabled` | Enable/disable OTel | Observability | `true` or `false` |
| `Serilog__MinimumLevel` | Log level | Serilog | `Information` or `Debug` |
| `Stripe__SecretKey` | Stripe API key (Phase 4 consumer-wired) | Billing | (secret) |

## 4. Migration Runner

### Today: Inline Migrations (Phase 1)

Migrations are applied automatically on startup via `DbContext.Database.MigrateAsync()`.

```csharp
// In SaasBuilder.Host.Program or sample host
public void ConfigureServices(IServiceCollection services)
{
    services.AddDbContext<SaasBuilderDbContext>(opts =>
    {
        opts.UseNpgsql(config.GetConnectionString("SaasBuilder"));
    });
}

public async Task Configure(IApplicationBuilder app)
{
    using (var scope = app.ApplicationServices.CreateScope())
    {
        var dbContext = scope.ServiceProvider.GetRequiredService<SaasBuilderDbContext>();
        await dbContext.Database.MigrateAsync();
    }
    
    // ... rest of configuration
}
```

### Future: `saas migrate` CLI (Phase 9)

```bash
saas migrate --environment production
```

This will:
1. Apply pending migrations per module in dependency order
2. Use Postgres advisory locks for leader election (prevents concurrent schema changes)
3. Roll back and alert on failure

### Health Probes

The SDK provides three distinct health endpoints:

| Endpoint | Purpose | Check |
|----------|---------|-------|
| `/health/live` | **Liveness** — is the process alive? | Process running, not deadlocked |
| `/health/ready` | **Readiness** — can it accept traffic? | DB connection OK, migrations complete |
| `/health/startup** | **Startup** — has initialization completed? | Modules loaded, dependencies wired |

**Kubernetes probe mapping:**

```yaml
livenessProbe:
  httpGet:
    path: /health/live
    port: 5000
  initialDelaySeconds: 30
  periodSeconds: 10
  timeoutSeconds: 5
  failureThreshold: 3

readinessProbe:
  httpGet:
    path: /health/ready
    port: 5000
  initialDelaySeconds: 20
  periodSeconds: 5
  timeoutSeconds: 5
  failureThreshold: 3

startupProbe:
  httpGet:
    path: /health/startup
    port: 5000
  initialDelaySeconds: 0
  periodSeconds: 5
  failureThreshold: 30  # 2.5 minutes grace for cold start
```

## 8. Database Setup

### RLS Migration Flow

Migrations under `migrations/rls/` enable row-level security:

```sql
-- migrations/rls/001_enable_rls.sql
ALTER TABLE products ENABLE ROW LEVEL SECURITY;

CREATE POLICY products_isolation ON products
  USING (tenant_id = current_setting('app.current_tenant_id')::uuid)
  WITH CHECK (tenant_id = current_setting('app.current_tenant_id')::uuid);

GRANT ALL ON products TO saas;
```

**Apply via EF Core:**

```csharp
protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    modelBuilder.Entity<Product>()
        .ToTable(t => t.ExcludeFromMigrations()); // Migration applied manually
    
    // OR use HasCheckConstraint/HasGeneratedQueryFilter
}
```

### FORCE RLS (Owner Bypass Protection)

Prevent the table owner from bypassing RLS:

```sql
-- In Postgres 10+
ALTER TABLE products FORCE ROW LEVEL SECURITY;
```

This forces RLS even for the table owner (recommended for production).

### Advisory Lock Pattern (Migration Safety)

The migration runner uses Postgres advisory locks to prevent concurrent migrations:

```csharp
// In migration job
const long advisoryLockId = 0xdeadbeef;
using var connection = new NpgsqlConnection(connectionString);
await connection.OpenAsync(ct);

var command = new NpgsqlCommand($"SELECT pg_advisory_lock({advisoryLockId});", connection);
await command.ExecuteScalarAsync(ct);

try
{
    // Apply migrations while holding lock
    var migrator = new Migrator(dbContext);
    await migratorApplyPendingMigrationsAsync(ct);
}
finally
{
    // Release lock
    command = new NpgsqlCommand($"SELECT pg_advisory_unlock({advisoryLockId});", connection);
    await command.ExecuteScalarAsync(ct);
}
```

## 11. Backups & Disaster Recovery

### Postgres Logical Backup

**Single daily backup:**

```bash
#!/bin/bash
BACKUP_DATE=$(date +%Y%m%d_%H%M%S)
BACKUP_FILE="/backups/saasbuilder_${BACKUP_DATE}.sql"

pg_dump \
  -h db.example.com \
  -U saas \
  -d SaasBuilder.Sample \
  > $BACKUP_FILE

gzip $BACKUP_FILE
aws s3 cp "${BACKUP_FILE}.gz" s3://my-backups/saasbuilder/
```

**Restore from backup:**

```bash
gunzip -c /backups/saasbuilder_20260510_120000.sql.gz | \
  psql -h db.example.com -U saas -d SaasBuilder.Sample
```

### Schema Changes (Expand-Migrate-Contract)

For breaking schema changes without downtime:

1. **Expand:** Deploy new app code that reads old AND new schema columns.
2. **Migrate:** Run backfill job to populate new columns from old data.
3. **Contract:** Deploy new app code that only reads new columns. Old columns can be dropped in a later deploy.

Example:

```csharp
// Phase 1: Expand
// App reads both old_email and new_email
var email = user.NewEmail ?? user.OldEmail;

// Phase 2: Migrate (background job)
UPDATE users SET new_email = old_email WHERE new_email IS NULL;

// Phase 3: Contract (next deploy)
// App reads only new_email
var email = user.NewEmail;
// Later: drop old_email column
```

## 15. Ops Runbooks

### Common Scenarios

#### Scenario: User Locked Out of Account

1. **Impersonate the user** (admin endpoint):
   ```bash
   curl -X POST https://api.example.com/api/v1/admin/impersonate \
     -H "Authorization: Bearer <admin-token>" \
     -H "Content-Type: application/json" \
     -d '{ "userId": "user-id", "reason": "Account locked — forcing password reset" }'
   ```

2. **Trigger password reset email** (or use Phase 9 CLI):
   ```bash
   saas admin force-reset-password --user-id=user-id
   ```

#### Scenario: Refund a Payment

1. **Check subscription status:**
   ```bash
   curl https://api.example.com/api/v1/billing/subscriptions/<tenant-id> \
     -H "Authorization: Bearer <token>"
   ```

2. **Process refund in Stripe** (or via Phase 6 admin API when available):
   ```bash
   curl https://api.stripe.com/v1/refunds \
     -u sk_live_...: \
     -d charge=ch_...
   ```

3. **Audit log** should record refund event automatically.

#### Scenario: Webhook Delivery Stuck

1. **Check delivery log** (Phase 5 Webhooks module):
   ```bash
   curl https://api.example.com/api/v1/webhooks/deliveries \
     -H "Authorization: Bearer <token>"
   ```

2. **Replay failed delivery:**
   ```bash
   curl -X POST https://api.example.com/api/v1/webhooks/deliveries/<id>/replay \
     -H "Authorization: Bearer <token>"
   ```

#### Scenario: Tenant Upgrade to Enterprise (SiloedDatabase)

1. **Phase 3 onwards:** Create a new tenant-specific database.
2. **Migrate data** via `saas migrate --tenant=enterprise-customer`.
3. **Switch connection string** via `ITenantResources` lookup.
4. **Verify RLS policies** on new database.
